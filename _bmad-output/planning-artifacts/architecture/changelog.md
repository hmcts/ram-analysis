| **v2.1 — Drop custom idempotency-key tables; use DB-native retry-safety primitives** | 2026-05-06 | Per-service `*_idempotency_keys` tables (`booking_idempotency_keys`, `payment_idempotency_keys`) and the per-service `IdempotencyFilter` boilerplate were dropped. Retry safety (FR45 — no double payment submission; FR30 — Booking creation safe to retry) is now provided by **PostgreSQL + JPA native primitives**: (a) **natural-key unique constraints** (`uq_bookings_vacancy_judge_session_date_type`, `uq_payments_cycle_run_date`, etc.) at the DB layer translate retry-of-successful-create into `409 Conflict`; (b) **JPA `@Version` optimistic locking** (`version integer NOT NULL DEFAULT 0` column on every domain entity) translates lost-update conflicts into `412 Precondition Failed`; (c) **PostgreSQL pessimistic row locking** (`SELECT … FOR UPDATE` via Spring Data JPA `@Lock(PESSIMISTIC_WRITE)`) on related rows handles cross-row workflow safety (e.g. Booking holds a row lock on the target vacancy across the booking-create-and-mark-filled transaction). The HTTP `Idempotency-Key` header is reserved as a future escape hatch but not implemented at MVP. **Why this matters:** the user called out that DB-level row locking + audit trail were already covering the ground that custom idempotency-key tables aimed at, and `*_idempotency_keys` tables were over-engineering for a problem PostgreSQL solves natively. **What changed:** [`./conventions.md`](./conventions.md) "Idempotency-key handling" section rewritten as "Retry safety and concurrency control" with the three native mechanisms; service-prefix table-name examples updated; constants example updated.
